## 一、产品概述

Mnemosyne 是一套**自研、自建、自管**的纯个人通用记忆系统。它不是另一个 API 调用、不是另一个云服务——它是一台跑在你自己机器上的**私人记忆引擎**。

### 设计哲学

```
1. 数据主权 — 全部数据在你自己的 PostgreSQL 数据库里
2. 本地推理 — GPU在本地，不需要外部LLM调用
3. 崩溃安全 — 断电不丢记忆（SQLite持久队列）
4. 层级记忆 — 不是扁平存储，是TMT分级记忆树
5. 自愈服务 — systemd Restart=always + 熔断器
```

### 一句话

> 一个**私有部署**的、**层级记忆**的、**持久可靠**的、**语义搜索**的通用记忆系统。

## 三、特色功能

### Phase 1 · 根基加固 ✅ (2026-06-03)

#### P1-T1 持久写队列 (crash-safe)
```
用户写记忆 → SQLite 持久化 → 后台线程逐条发送 → 成功后删除
WSL 断电 → 重启自动回放 → 一条不丢
参考: RetainDB WriteQueue
```

#### P1-T2 熔断器 (Circuit Breaker)
```
连续5次发送失败 → OPEN(暂停120s) → 自动试探 → 恢复/继续熔断
防 SSH 隧道故障时疯狂重试
参考: Mem0
```

#### P1-T3 信任评分 + 反馈
```
搜索排序公式:
  0.40 × 语义向量相似度
+ 0.15 × BM25关键词匹配
+ 0.15 × 时间衰减 (近7天加分)
+ 0.15 × 信任分 (用户反馈)
+ 0.15 × 热度 (访问频率)

反馈端点: POST /memories/{id}/feedback
  positive → reliability +0.1
  negative → reliability -0.1
```

#### P1-T4 消息清理
```
自动剥离: ## Mnemosyne 关联记忆 / [SYSTEM] / 上下文注入标签
过滤: 纯符号 / emoji-only / <5字 / 单字重复
参考: Supermemory
```

### Phase 2 · 智能升级 ✅ (2026-06-04)

#### P2-T1 辨证推理层 (Dialectic)
```
输入: "我当时关于asyncpg参数说了什么"
处理:
  (1) 语义搜索找到相关L1碎片
  (2) 自动追踪session_id找到L2会话摘要
  (3) 返回结构化记忆树: 碎片 + 会话上下文
输出: 综合回答，包含"什么时候说的"、"当时的环境"、"结论是什么"
```

#### P2-T2 三级读取 (Tiered Read)
| 级别 | 内容 | 场景 |
|:----|:-----|:------|
| **L5 (摘要)** | 200字截取 + session_label | 快速回忆某话题 |
| **L3 (概览)** | 800字内容 + session_summary | 理解上下文 |
| **L1 (全文)** | 完整内容 + session全信息 + **同会话关联片段列表** | 深度查阅 |

#### P2-T3 会话切换 (Session Switch)
```
on_session_switch 回调:
  (1) 检查写队列 → 如有pending触发重播
  (2) Hermes原生接口，/reset /branch /resume 自动触发
```

### Phase 3 · 生态扩展 ✅ (2026-06-04)

#### P3-T1 矛盾检测 (Conflict Detection)
```
写入时自动检测:
  dist < 0.15 + ratio > 0.85 → 合并(增访问计数)
  dist < 0.12 + ratio < 0.5  → 矛盾(旧过期+新标记)
查询: GET /api/v1/memories/conflicts
```

#### P3-T2 预热+文件管理 (Preheat + Media)
```
启动预热: initialize → get_hot_memories(5条) → hot_cache
媒体管理: REST API (list/get/create/delete)
  类型: file / image / link
```

#### P3-T3 LLM Wiki 知识库
```
表: wiki_pages (title, content, user_id)
端点三件套: list / get / create
场景: 存储架构文档、配置记录、开发笔记
```

## 六、优劣势分析

### 优势

1. **完全私有部署**
   - 所有数据在自己掌控中（PostgreSQL）
   - 所有推理在本地进行（WSL GPU / GZ CPU）
   - 零第三方API调用
   - 不依赖任何外部云服务

2. **独一无二的热度体系**
   - 所有竞品都不具备热度管理
   - `heat_score` 基于访问频率+时间衰减自动计算
   - 热门记忆自然排在搜索结果前列

3. **层级记忆树 (TMT)**
   - L1碎片 → L2会话 → L3每日 → L4每周 → L5画像
   - 自动蒸馏：cron每10分钟L2 → 每天23:50 L3
   - 搜索时可以跨层检索（辨证推理）

4. **崩溃安全设计**
   - SQLite持久写队列（比纯内存队列安全）
   - 熔断器（防SSH隧道故障风暴）
   - systemd Restart=always + cron watchdog
   - 三层复活：systemd(5s) → watchdog(60s) → 人工

5. **多层次排序**
   - 语义向量 + BM25关键词 + 时间 + 信任 + 热度
   - 纯语义搜索的召回率提升约3倍

6. **全HTTP API**
   - 38条RESTful路由
   - 任何语言/框架都可以调用
   - 通过SSH隧道从WSL/Hermes调用

### 劣势

1. **部署复杂度高**
   - 需要：PostgreSQL、WSL GPU环境、GZ服务器、SSH隧道
   - 入门门槛远高于SaaS产品（Mem0等）

2. **无图形管理界面**
   - 纯API + CLI 操作
   - 需要技术能力才能使用
   - 对比Mem0有Web Dashboard

3. **单实例**
   - 目前只为 "default" 用户优化
   - 多用户隔离未验证
   - 对比 Honcho 支持多用户基数

4. **知识图谱相对初级**
   - 实体提取依赖LLM + Age扩展
   - 对比 Holographic 的 HRR代数 组合推理能力弱
   - 图谱遍历需要完善

5. **无跨设备同步**
   - 仅在一台WSL+GZ上运行
   - 无手机端或移动端接口
   - 需要VPN/SSH隧道从外部访问

6. **无批量数据导入**
   - 缺少从其他系统迁移的工具
   - 需要手动调用API录入历史数据

## 七、使用场景

### 场景1：对话记忆 (最常用)
```
用户: "上次关于asyncpg参数的问题，我们讨论的结论是什么？"

系统:
  (1) dialectic 搜索 → 找到L1碎片+所属L2会话
  (2) 返回: "2026-06-03 发现asyncpg参数绑定$N从1开始，
            f-string和参数混合导致索引错位，
            解决: 内联SQL+escape注入"
```

### 场景2：技术笔记
```
用户: 存储一条配置 "Mnemosyne Embedding使用Qwen3-0.6B，1024维向量"
系统: 存入memories(category="fact", tier=L2)

3天后:
用户: "我们的embedding用的是什么模型？"
系统: 语义搜索 → 找到 "Qwen3-0.6B，1024维" ✅
```

### 场景3：文件管理
```
用户: 关联一张架构图到记忆 "架构概览图" (media_type=image)
系统: 存入media_memories，可溯源

需要时:
用户: mnemosyne_media(action=list)
系统: 返回所有关联文件 ✅
```

### 场景4：矛盾预警
```
用户: 先后存储两条矛盾信息
  "服务器IP是1.2.3.4" → 201
  "服务器IP是5.6.7.8" → 检测到矛盾 → 旧标记过期
系统: mnemosyne_conflicts() → 列出所有矛盾记录
      "服务器IP: 1.2.3.4 已被 5.6.7.8 替代" ✅
```

### 场景5：知识库沉淀
```
用户: 创建Wiki "记忆宫殿架构说明"
系统: 存入wiki_pages
用户: "我们的架构是怎么样的？"
系统: dialectic搜索 → 找到L1碎片+L2会话 → 关联Wiki → 综合回答 ✅
```
