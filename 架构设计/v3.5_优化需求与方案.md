## 二、使用审计发现：核心问题

### 核心矛盾

| 维度 | 数据 | 解读 |
|:-----|:----|:------|
| 记忆总数 | **338 条** | 写入活跃 |
| 从未被访问 | **194 条 (57%)** | 写多读少 |
| 最近访问 | 101 条本周 / 22 条本月 | 访问集中在新数据 |
| 搜索调用 | 仅开发/调试时 | 日常对话无主动检索 |
| access_count | 全部为 0 | 无任何记忆被真实查询过 |

**现状：系统是一个「日记本」，不是一个「记忆助手」。** 存了很多但从未被主动调用。

### 根因

1. **Hermes memory provider 自动存储（nudge/flush）把对话记下来了，但对话中不会自动查**
2. **搜索入口太深**：需要用户主动说"查一下我说的XX"，不会自动关联
3. **session_id 利用率低**：大部分 memory 的 session_id=0，无法通过会话追踪

## 三、v3.5 优化方案

### P0 — 自动记忆召回 (Proactive Recall)

**目标**：对话中自动搜索相关旧记忆，在回复中引用——让系统从「被动存储」变成「主动记忆」

**方案**：

```
对话消息 → Hermes Provider on_message 回调
  → embedding 搜索最近 3 条相关记忆 (mnemosyne_search)
  → 结果注入到 system prompt 的 "相关记忆" 段
  → AI 回复时自动引用（如 "你说过之前也遇到过这个问题..."）
```

| 项目 | 细节 |
|:-----|:------|
| 触发时机 | 用户发送新消息时 |
| 搜索范围 | 全部记忆 (除当轮对话) |
| Top-K | 3 条最相关 |
| 注入位置 | System prompt 末尾 `[相关记忆]\n...` |
| 性能要求 | < 500ms (embedding 20ms + 搜索 10ms + 隧道 5ms) |
| 开关 | Hermes provider config 可配 `proactive_recall: true/false` |

**依赖**：`mnemosyne_search` endpoint 已就绪（权重已修正），无需新增 API。

### P1 — 记忆足迹补全

**目标**：`memory_traces` 目前只记录 feedback+superseded，补全 store/recall 追踪

**方案**：
- `store` 写入 trace（含 content 摘要 + category）
- `recall` 写入 trace（含 query + 返回条数）
- 新增 `GET /api/v1/memories/search-traces` 端点 → 按 query 搜索最近的检索记录

### P1 — 会话上下文关联 (session_id)

**目标**：让记忆能通过会话追踪，而不是孤立碎片

**方案**：
- Hermes Provider `on_session_switch` 回调已就绪，但 session_id 传递链路需要验证
- 在 `sync_turn` 时确保 `session_id` 参数正确传递到 GZ

### P2 — 版本管理规范

**目标**：上述修复完成后，打 tag + 推远程
