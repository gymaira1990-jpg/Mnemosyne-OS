## 二、核心发现：Mnemosyne 的位置

### 我们的独特优势（竞品没有）

| 能力 | 竞品状态 | 我们的状态 |
|:-----|:---------|:-----------|
| **TMT 记忆层级 (L1→L5)** | ❌ 无。Mem0/Letta只有单层 | ✅ 独有，从碎片→会话→每日→每周→画像 |
| **辨证推理 (Dialectic Reasoning)** | ❌ 无。Hindsight有Reflection，但整合弱 | ✅ 跨L1/L3/L5综合，带会话上下文 |
| **矛盾检测 (Conflict Detection)** | ❌ 无。竞品完全忽略 | ✅ 语义差异对比+可信度衰减 |
| **Wiki 知识库** | ❌ 无 | ✅ 版本化文档知识库 |
| **三级读取 (L5/L3/L1)** | ❌ 无 | ✅ 摘要→概览→全文分级 |

### 我们的关键差距（P0优先级）

| 差距 | 严重度 | 行业标准 | 我们的状态 |
|:-----|:------:|:---------|:-----------|
| **时序感知** | 🔴 P0 | Graphiti双时态建模，Mem0路线图中有 | ❌ 无 valid_time，无事实版本 |
| **实体提取质量** | 🔴 P0 | Mem0自动提取+实体链接，Graphiti消歧，Cognee六阶段 | ⚠️ 初级LLM提取，无消歧 |
| **可视化** | 🔴 P0 | Letta ADE三面板，Cognee UI，Obsidian图谱 | ❌ 纯API/CLI |
| **信念系统** | 🟡 P1 | Hindsight四网络(事实/经历/摘要/信念) | ⚠️ 有初步信念存储，无演化追踪 |
| **批量导入** | 🟡 P1 | 所有系统都有 | ❌ 无 |
| **评测基准** | 🟢 P2 | Mem0接入LoCoMo/LongMemEval/BEAM | ❌ 无 |

### 不做的事（基于调研的战略决策）

- ❌ **不引入 Neo4j/FalkorDB** — 新增基础设施运维成本 > 收益。PG+pgvector+AGE 足够支撑到百万级
- ❌ **不转型全栈 Agent Runtime**（Letta 路线）— 我们的定位是记忆层，不是Agent执行引擎
- ❌ **不依赖商业 SaaS**（Supermemory/Zep Cloud）— 保持自托管，100%数据控制
- ❌ **不从零做可视化** — 用 vis-network 快速出 MVP，而非手撸 D3/Three.js

## 三、v4.0 新路线（相比原方案的重大修正）

### 原方案（上次的 v4.0 升级方案）

```
Week 1: A+D 合并（架构修复+Excalidraw可视化）
Week 2: B（实体解析Pipeline）
```

### 新方案（基于深度调研）

```
P0 ─ 实体解析 Pipeline ⭐ 第一优先级
P0 ─ 时序版本化 ⭐ 第一优先级
P0 ─ 可视化 MVP (图谱视图) ⭐ 第一优先级
P1 ─ 信念系统增强
P2 ─ 评测基准 + 批量导入
```

**为什么修正？**
- 原方案把**可视化**排第一，但调研发现**实体解析**和**时序版本化**才是根本差距
- Excalidraw 不适合做主浏览视图（大数据量性能差），调整为编辑/标注工具
- 真正的浏览视图应先用 D3/vis-network 快速出

## 四、推荐路线：4 个 Phase

### Phase A — 实体解析 Pipeline（1-2 周）

```
当前: LLM prompt 提取（初级）
目标: 三级级联 Pipeline

第1层(spaCy, ~0.5ms): 快速NER + 词性标注 → 过滤停用词
第2层(GLiNER, ~10-50ms): 零样本NER → 多种关系类型
第3层(本地LLM, ~1-3s): 验证 + 关系抽取 + 实体消歧

存储: 修复 AGE Schema（统一 Entity 节点 + RELATES 关系边）
API: POST /api/v1/memories/{id}/entities 触发提取
```

### Phase B — 时序版本化（1 周）

```
方案: SCD Type 2 + temporal_tables 扩展（零外部依赖）

核心表变更:
  memories 表增加 valid_from / valid_to / version / changed_by
  写入自动触发历史归档（触发器实现）

查询模式:
  某时间点快照: valid_from <= T AND valid_to > T
  完整编辑历史: (current) UNION ALL (history) ORDER BY version
  矛盾检测升级: 自动标记时序矛盾（如旧事实被新事实覆盖）
```

### Phase C — 可视化 MVP（2 周）

```
MVP: 图谱视图 + CRUD + 搜索
技术栈: React + Tailwind + vis-network（6KB gzip）
部署: 独立 React App → 后续集成到 Hermes Web UI

视图优先级:
  Phase C1 (1周): 图谱视图（节点=实体，边=关系，大小=热度，颜色=分类）
  Phase C2 (1周): 搜索+详情面板+CRUD+反馈按钮

Excalidraw 定位: 编辑/标注工具，非主浏览视图
```

### Phase D — 存储与性能优化（持续）

```
立即（今天）:
  ① 创建 HNSW 索引（零风险，立刻见效）
  ② 调优 postgresql.conf（shared_buffers=512MB 等）
  ③ 集成 HotMemoryCache（替换简单 dict）

短期（1周内）:
  ④ 批量写入缓冲（10条/flush，降 SSH 开销）
  ⑤ ANN 粗筛 → Reranker 精排流水线

中期（1月内）:
  ⑥ 评估 PG 18 迁移（AIO + Skip Scan）
  ⑦ 五维修序下推到 PG 单条 SQL
  ⑧ 按时间分区 + 备份自动化
```

## 五、关键架构决策

| 决策 | 选择 | 理由 |
|:-----|:-----|:------|
| 向量索引 | HNSW > IVFFlat | 增量写入天然支持，召回率95% |
| 向量维度 | 768 halfvec（节省50%空间） | Qwen3-0.6B输出768d，1523d仅提升2%召回 |
| 时序方案 | SCD Type 2 + 触发器 | 零外部依赖，纯PG实现 |
| 缓存 | Python LRU+TTL → PG shared_buffers | 不需要Redis/Memcached |
| 可视化 | vis-network(首批) → react-force-graph(规模) | 6KB gzip，快速启动 |
| Excalidraw | 编辑/标注工具，非主浏览 | 大数据量性能瓶颈 |
| 连接池 | asyncpg min=2 max=8 | 轻量服务器，uvloop加速 |
| 备份 | pg_dump(每日) + WAL归档(PITR) | 腾讯云轻量够用 |

## 七、一句话总结

> **Mnemosyne 不需要做成另一个 Mem0 或 Zep。** 我们的 TMT 层级 + 辨证推理 + 矛盾检测已经是竞品没有的护城河。v4.0 的目标是补齐「实体解析 + 时序版本化 + 可视化」三块短板，同时牢牢守住我们独有的记忆层级体系。不做大而全，做差异化完整的 Agent Memory 产品。
