## 二、外部工具调研

### 工具一：Excalidraw AI 🏆 重点采用

| 维度 | 评估 |
|:----|:------|
| **开源** | ✅ 完全开源 (MIT) |
| **MCP支持** | ✅ 原生 MCP 工具集 (18个工具) |
| **核心能力** | 自然语言 → 手绘风格图表的 AI 生成 |
| **技术栈** | TypeScript + React，纯前端 |
| **集成难度** | 🟢 低 — 直接通过 MCP 协议接入 Hermes |
| **我们用途** | 记忆图谱可视化前端、架构图生成、关系图展示 |

**适合我们**：不需要自己写前端，直接复用 Excalidraw MCP 工具做可视化

**不适合**：不能做图数据库后端（只做可视化）

### 工具二：Knowledge Graph Studio (WhyHow)

| 维度 | 评估 |
|:----|:------|
| **开源** | ✅ 开源 (具体 license 待确认) |
| **核心能力** | 规则化实体解析、模块化图构建、API-first |
| **技术栈** | MongoDB + FastAPI + OpenAI |
| **集成难度** | 🟡 中 — 需要跑额外服务 + MongoDB |
| **我们用途** | 可借鉴实体解析规则、三元组构建模式 |

**适合我们**：Rule-based entity resolution 的设计思路值得借鉴
**不适合**：MongoDB 底座与我们 PG 架构不匹配，整套部署太重

### 工具三：Graphiti (GetZep) 🏆 重点研究

| 维度 | 评估 |
|:----|:------|
| **开源** | ✅ 完全开源 |
| **核心能力** | **时序知识图谱** (Temporal Context Graph) |
| **技术栈** | Python + 支持 FalkorDB/Neo4j |
| **MCP支持** | ✅ 有专用的 MCP Server |
| **亮点** | 追踪事实随时间变化、维护来源溯源、混合语义+图谱检索 |
| **集成难度** | 🟡 中 — 额外图数据库依赖 |

**适合我们**：时序知识图谱的"事实追踪变化"设计与我们的 conflict detection 理念高度一致
**不适合**：需要额外图数据库（Neo4j/FalkorDB）

### 工具四：Zep (Graphiti 的企业版)

| 维度 | 评估 |
|:----|:------|
| **开源** | ⚠️ 社区版开源，企业版付费 |
| **核心能力** | 企业级 Agent Memory，百万级 Context Graph |
| **我们用途** | 仅做参考，不直接集成 |

**评估**：太重了，企业级产品。但它的论文《Zep: A Temporal Knowledge Graph Architecture for Agent Memory》中提出的 **Episode→Entity→Semantic Edge** 三层架构值得学习。

## 三、四套升级方案

### 方案 A：轻量化 — Excalidraw + PG 增益（推荐 🔥）

**原则**：不动数据库底座，只加外部可视化和管理工具

```
Hermes Agent
├── MCP Bridge (现有)
│   ├── mnemosyne_mcp.py       ← 保持
│   └── tmt_mcp.py             ← 保持
├── + Excalidraw MCP           ← 新增：直接复用
│     可视化知识图谱、架构图、记忆树
└── + Obsidian 集成            ← 新增：可选
     Markdown 笔记 ←→ Mnemosyne Wiki 双向同步
```

**改动量**：极小（~50行配置）
**风险**：🟢 极低
**价值**：立即获得可视化能力

### 方案 B：中等 — 实体解析 Pipeline（推荐 🔥）

**原则**：自研实体解析层，不引入外部图数据库

```
写入记忆时:
  1. detect_conflict (已有) → 合并/冲突检测
  2. + entity_extract (LLM) → 提取实体+"主谓宾"三元组
  3. + entity_resolve (规则) → 同实体合并（参考WhyHow规则引擎）
  4. + age_graph (PG Age) → 存入Age知识图谱（已有！不额外引入DB）

查询时:
  1. 语义搜索 (已有) → 找到相关L1碎片
  2. + 图谱遍历 (age_graph) → 沿实体关系展开邻居节点
  3. + tiered_read (已有) → 获取关联片段全文
```

**改动量**：中（~300行）
**风险**：🟡 中
**价值**：真正的知识图谱能力，数据在已有 PG 中

### 方案 C：重型 — 引入 Graphiti/Zep

**原则**：引入外部图数据库和图引擎

```
Mnemosyne (PG + pgvector) ─── 记忆存储主力
    ↑ 语义搜索/热度/信任
Graphiti (FalkorDB/Neo4j) ─── 知识图谱侧翼
    ↑ 时序事实追踪/实体关系
```

**改动量**：大（额外DB + 学习曲线）
**风险**：🔴 高
**价值**：获得企业级图谱能力，但运维成本飙升

### 方案 D：极简 — 只改现有架构

**原则**：不引入任何新工具，纯代码优化

```
1. 修复 tmt_tree_edges 类型混用 (bigint+UUID→统一UUID)
2. 增加实体提取到 metadata 字段
3. 增加"相似记忆推荐"端点 (利用已有向量搜索)
4. 增加记忆关系手动标注端点
```

**改动量**：小（~100行）
**风险**：🟢 极低
**价值**：立即可用，但缺乏突破性提升

```
Phase 6.0: 前端 UI
```

## 五、Excalidraw 集成方案

### 接入方式

```
Hermes Agent 原生支持 MCP → Excalidraw 有 MCP 工具集

1. 安装: npm install -g @excalidraw/excalidraw-cli
   或: 直接通过 Web UI 的 /excalidraw 命令

2. 使用: 
   /excalidraw "把最近关于asyncpg的记忆画成关系图"

   → 画出：记忆#193 → 问题"asyncpg参数" → 结论"内联SQL"
```
